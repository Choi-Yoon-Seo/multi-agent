# Week 09. 수업 실습: 멀티에이전트 설계 예시

## 1. 도메인과 작업

- 도메인: 학과 공지 정리
- 작업: 여러 공지를 읽고 학생 유형별 행동 목록 만들기
- 이 작업이 반복되는 상황: 수강 신청, 장학금, 졸업 요건, 교환학생 신청 공지가 동시에 올라올 때
- 최종 사용자가 원하는 결과: 1학년, 4학년, 복수전공생, 교환학생 준비생별로 해야 할 일과 마감일을 바로 확인한다

## 2. 단일 에이전트로 부족한 이유

1. 공지의 사실 추출, 학생 유형 분류, 최종 안내문 작성이 한 응답에 섞인다.
2. 마감일이나 대상 학년을 잘못 읽어도 스스로 검증하지 못할 수 있다.
3. 학생에게 안내하기 전, 중요한 마감일과 신청 조건은 사람이 확인해야 한다.

## 3. 역할 설계

| 역할 | 입력 | 작업 | 출력 | 검증 기준 |
|---|---|---|---|---|
| 공지 수집자 | 공지 원문 | 대상, 마감일, 준비물, 링크 추출 | 공지별 사실 목록 | 각 항목에 출처가 있는가 |
| 대상 분류자 | 공지별 사실 목록 | 학생 유형별 관련 공지 분류 | 유형별 후보 목록 | 대상 조건이 맞는가 |
| 안내 작성자 | 유형별 후보 목록 | 학생이 바로 볼 안내문 작성 | 유형별 할 일 목록 | 행동, 마감일, 링크가 포함됐는가 |
| 검토자 | 최종 안내문 + 사실 목록 | 빠진 조건과 위험 문장 확인 | 수정 요청 또는 승인 | 원문에 없는 내용이 없는가 |

## 4. 처리 흐름

```text
[사용자 입력: 공지 여러 개]
   ↓
[공지 수집자]
   ↓
[대상 분류자]
   ↓
[안내 작성자]
   ↓
[검토자]
   ↓
[사람 승인]
   ↓
[최종 출력: 학생 유형별 할 일 목록]
```

## 5. 공유 상태 초안

- user_request: 사용자가 원하는 정리 방식
- notice_sources: 공지 원문과 링크
- extracted_facts: 공지별 대상, 마감일, 준비물, 링크
- student_groups: 1학년, 4학년, 복수전공생, 교환학생 준비생
- grouped_tasks: 학생 유형별 후보 작업
- draft_guide: 안내 작성자가 만든 초안
- review_notes: 검토자의 수정 요청
- human_feedback: 사람이 승인하거나 고친 내용
- final_answer: 최종 안내문

## 6. 사람 승인 지점

- 승인 지점: 검토자가 최종 안내문을 확인한 뒤
- 사람이 확인해야 하는 이유: 마감일, 신청 자격, 링크 오류는 학생에게 직접 피해를 줄 수 있다
- 승인 후 다음 단계: 최종 안내문을 학과 홈페이지나 공지 채널에 맞게 다듬는다

## 7. 실패 조건

1. 공지가 1개뿐이고 단순 전달이면 단일 에이전트가 더 낫다.
2. 공지 수집자가 마감일을 잘못 추출하면 이후 모든 역할의 결과가 틀어진다.
3. 대상 분류자가 복수전공생과 전과생 조건을 혼동하면 잘못된 안내가 나간다.
4. 검토 없이 자동 게시하면 원문에 없는 조건이 추가될 수 있다.

## 8. 멀티에이전트를 쓰는 근거

- 사실 추출, 대상 분류, 안내문 작성, 검토의 책임이 서로 다르다.
- 각 단계의 입력과 출력이 표로 분리된다.
- 사람이 승인해야 하는 지점이 분명하다.

## 9. 5문장 회고

1. 이 작업은 단순 요약보다 대상 분류와 검증이 중요하다.
2. 역할을 나누면 공지 원문에서 뽑은 사실과 최종 안내문을 분리할 수 있다.
3. 가장 중요한 공유 상태는 마감일과 신청 대상이다.
4. 공지가 적거나 조건이 단순하면 멀티에이전트가 오히려 느리다.
5. 10주차 Supervisor 패턴에서는 검토자가 수정 요청을 보낼 때 어느 역할로 되돌릴지 결정할 수 있다.
