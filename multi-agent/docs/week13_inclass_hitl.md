# Week 13. 수업 실습: HITL 승인 예시

## 1. 예시 과제

- 기반: 12주차 RAG 멀티에이전트
- 추가 기능: 승인 노드, rewrite 분기, 비용 로그
- 목표: 모델 검증 이후에도 사람이 최종 결정을 내리게 한다.

## 2. 승인 노드 설계

| 항목 | 내용 |
|---|---|
| 승인 전 보여줄 정보 | draft, 출처 청크 일부, grounding 결과 |
| approve 입력 | 최종 답변으로 종료 |
| rewrite 입력 | 사람 피드백을 남기고 writer로 되돌림 |
| 거부 이유 기록 | messages 또는 review_notes에 저장 |

## 3. 예상 흐름

```text
[질문]
   ↓
[검색]
   ↓
[작성자]
   ↓
[출처 검증 게이트]
   ↓ PASS
[사람 승인]
   ├─ approve → [최종 답변]
   └─ rewrite: 피드백 → [작성자]
```

## 4. 비용 로그

| 항목 | 기록 |
|---|---|
| 호출 노드 | search / writer / grounding / approval |
| 입력 토큰 | 모델 응답에서 확인 가능한 경우 기록 |
| 출력 토큰 | 모델 응답에서 확인 가능한 경우 기록 |
| 응답 시간 | 노드별 측정 |
| 비용 절감 후보 | 반복 호출이 많은 노드 |

## 5. approve 예시

- 사람 입력: `approve`
- 기대 결과: 최종 답변으로 종료
- 기록: `[승인] 사람 승인 통과`

## 6. rewrite 예시

- 사람 입력: `rewrite: 마감일과 출처를 더 분명히 표시`
- 기대 결과: writer로 돌아가 초안을 다시 작성
- 기록: `[거부] 마감일과 출처를 더 분명히 표시`

## 7. 5문장 회고

1. HITL은 모델이 자동으로 끝내면 위험한 지점에 사람 결정을 넣는 방식이다.
2. 출처 검증이 PASS여도 표현이나 책임 문제는 사람이 볼 수 있다.
3. rewrite 분기는 사람 피드백을 시스템 상태에 남기고 다시 작성하게 한다.
4. 비용 로그를 보면 어느 노드가 가장 자주 또는 오래 호출되는지 알 수 있다.
5. 최종 프로젝트에서는 HITL 지점을 한 곳 이상 명시해야 한다.
