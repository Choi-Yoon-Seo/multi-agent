## 멀티에이전트 설계표

- 도메인: 내 취향 플레이리스트 생성

- 작업: 현재 상황 한 줄과 선호 장르·아티스트 필터를 입력받아, 플레이리스트 초안을
  순차적으로 생성하고 사람 승인을 거쳐 Spotify 저장 준비 완료 상태로 만든다

- 단일 에이전트로 부족한 이유 (3줄):
  1. 상황 해석(BPM·에너지 변환), 필터 교집합 계산, Spotify API 검색, 순서 배치가 한 응답에 섞이면
     각 단계의 오류가 어디서 발생했는지 추적할 수 없다.
  2. 자기가 구성한 플레이리스트를 자기가 검증하면 BPM 연속성 위반이나 아티스트 편중 같은
     규칙 오류를 스스로 잡지 못한다.
  3. 필터를 바꿔 재시도하고 싶을 때 어느 단계가 원인인지 추적할 수 없고, 사람이 필터를
     확정하는 지점이 없어 AI가 혼자 방향을 결정하게 된다.

- 역할 1:
  - 이름: 상황 분석 에이전트
  - 입력: 사용자가 선택한 상황 (드라이브·집중·슬플 때·샤워·운동 등)
  - 작업: 상황을 매핑 테이블 기반으로 음악 속성 4개로 변환
  - 출력: { bpm_range, energy, valence, instrumental }
  - 검증 기준: 4개 필드 모두 채워졌는가 / 테이블 미등록 상황이면 사용자에게 확인 요청

- 역할 2:
  - 이름: 필터 병합 에이전트
  - 입력: 역할 1의 음악 속성 딕셔너리 + 사용자가 선택한 장르·아티스트 필터
  - 작업: 상황 속성과 사용자 필터를 교집합으로 병합해 Spotify 검색 파라미터 생성
  - 출력: Spotify 검색 파라미터 { seed_genres, seed_artists, target_bpm, target_energy, 
    target_valence }
  - 검증 기준: 파라미터 유효성 검사 통과 / 충돌 감지 시 자동 진행 금지

- 역할 3:
  - 이름: 플레이리스트 구성 에이전트
  - 입력: Spotify API 후보곡 20~30개
  - 작업: 도입→전개→정점→마무리 흐름으로 순서 배치 및 규칙 검증
  - 출력: 순서 확정된 플레이리스트 초안 10~20곡 + 각 곡의 선택 이유 한 줄 요약
  - 검증 기준: BPM 연속성 규칙 통과 / 최소 10곡 이상 / 단일 아티스트 비중 40% 이하

- 처리 흐름:
  [사용자 입력: 상황 선택 + 장르·아티스트 필터]
     ↓
  [상황 분석 에이전트]
     ↓
  [필터 병합 에이전트] ←─────────────────────┐
     ↓                                   │ 필터 재조정
  [Spotify API 검색]                      │
     ↓                                   │
  [플레이리스트 구성 에이전트]                  │
     ↓ (검증 실패 시 사람에게 필터 재조정 요청 후 역할 2 재실행)
  [사람 승인: 최종 확인 — 저장할지·필터 재조정할지·전면 재시작할지]
     ↓
  [Spotify 저장 또는 JSON 내보내기 완료]

- 공유 상태:
  - context         : 사용자가 선택한 상황
  - music_attrs     : 상황 분석 에이전트가 변환한 음악 속성
  - filters         : 사용자가 입력한 장르·아티스트 필터
  - search_query    : 필터 병합 에이전트가 생성한 Spotify 검색 파라미터
  - candidate_songs : Spotify API 검색 결과 후보곡
  - playlist_draft  : 플레이리스트 구성 에이전트가 만든 초안
  - approval_status : 저장할지·필터 재조정할지·전면 재시작할지 (사람 승인 결과)
  - final_outputs   : 저장 확정본

- 사람 승인 지점:
  - 승인 (플레이리스트 구성 직후): 초안과 각 곡의 선택 이유를 보고 저장 여부를 결정한다.
    취향은 정답이 없고 같은 조건이라도 그날 기분에 따라 결과가 달라지므로
    AI가 제안하고 내가 선택하는 구조로 최종 판단을 유지한다.
  - 옵션: ① 저장 진행 ② 필터 재조정(장르·아티스트만 바꿔 역할 2 재실행)
         ③ 전면 재시작(상황부터 다시)

- 멀티에이전트가 더 나아질 것이라 보는 근거:
  - 상황 해석·필터 병합·순서 배치의 책임이 서로 달라 역할을 합치면 오류 원인 추적이 불가능하다.
  - 각 역할의 검증 기준이 BPM 범위·곡 수처럼 수치로 떨어져 입출력이 표로 분리된다.
  - 사람 승인 명분이 명확하다: 최종 저장 결정.
  - 검증 실패 시 필터 병합으로 돌릴지 상황 분석으로 돌릴지를 Supervisor가 동적으로 결정.

- 멀티에이전트를 쓰면 안 되는 조건:
  1. Spotify API 없이 LLM이 곡명을 직접 생성하는 경우 — 단일 에이전트가 빠르다.
  2. 상황이 단 1~2개로 고정되어 매핑 테이블이 필요 없는 경우 — 구조화된 단일 프롬프트로 충분하다.
  3. 사용자 필터(장르·아티스트)가 없는 경우 — 필터 병합 에이전트가 판단 기준이 없어진다.
  4. 지금 당장 빠르게 1개만 만들고 싶은 경우 — 3단계 파이프라인이 오히려 마찰이 된다.
