## 멀티 에이전트 설계표

- 도메인: 내 취향 플레이리스트 생성

- 작업: 상황(드라이브·집중·슬플 때 등)을 선택하고 선호 장르·아티스트 필터를 입력 
  받아, 음악 속성으로 변환 → Spotify 검색 → 플레이리스트 순서 구성 → 사람 승인 → 최종 저장

- 단일 에이전트로 부족한 이유 (3줄):
  1. 상황 해석(BPM·에너지 변환), 필터 교집합 계산, Spotify API 검색, 순서 배치가 
     한 응답에 섞이면 각 단계의 오류가 어디서 발생했는지 추적할 수 없다.
  2. 사람이 "필터만 바꿔 재시도"하고 싶을 때 단일 에이전트는 상황 해석부터 전부. 재실행해야 하지만, 
     멀티 에이전트는 필터 병합 단계부터 재실행할 수 있다.
  3. 자기가 구성한 플레이리스트를 자기가 검증하면 BPM 연속성 위반이나 아티스트 
     편중 같은 규칙 오류를 스스로 잡지 못한다.

- 역할 1: 상황 분석기
  - 입력: 사용자가 선택한 상황 텍스트 (드라이브/ 집중/ 슬플 때/ 샤워/ 운동 등)
  - 작업: 상황을 매핑 테이블 기반으로 음악 속성 4개(BPM 범위·에너지·발렌스
    ·인스트루멘탈 여부)로 변환. 테이블에 없는 상황은 가장 유사한 상황으로. 매핑한 뒤 사용자에게 확인
  - 출력: { bpm_range, energy, valence, instrumental }
  - 검증 기준: 4개 필드 모두 채워졌는지 확인 / 테이블 미등록 상황이면 매핑. 결과를 사용자에게 
    노출하고 진행 여부 선택 요청

- 역할 2: 필터 병합기
  - 입력: 역할 1의 음악 속성 딕셔너리 + 사용자가 선택한 장르(KPop·JPop·Pop. 등)·아티스트 필터
  - 작업: 상황 속성과 사용자 필터를 교집합으로 병합해 Spotify API 검색 파라미터 
    생성. 충돌 감지 시(예: 슬플 때 + 댄스 장르) 경고 후 우선순위 선택 요청
  - 출력: Spotify 검색 파라미터 { seed_genres, seed_artists, target_bpm, 
    target_energy, target_valence, min/max 범위 }
  - 검증 기준: 파라미터 유효성 검사 통과 / 충돌 감지 시 자동 진행 금지 / 필터가. 없으면 상황     
    속성만으로 진행

- 역할 3: 플레이리스트 구성기
  - 입력: Spotify API 후보곡 20~30개
  - 작업: 도입 → 전개 → 정점 → 마무리 흐름으로 순서 배치. 연속된 두 곡의. BPM. 차이 30 이하 
    유지, 같은 아티스트 3곡 연속 방지, 단일 아티스트 비중 40% 이하 제한
  - 출력: 순서 확정된 플레이리스트 초안 10~20곡 + 각 곡의 선택 이유 한 줄 요약
  - 검증 기준: BPM 연속성 규칙 통과/ 최소 10곡 이상/ 아티스트 다양성 조건 충족

- 처리 흐름:
  [사용자: 상황 선택 + 장르·아티스트 필터 입력]
     ↓
  [역할 1: 상황 분석기]
     ↓
  [역할 2: 필터 병합기] ←──────────────────┐
     ↓                                 │ 필터 재조정
  [Spotify API 검색]                    │
     ↓                                 │
  [역할 3: 플레이리스트 구성기]              │
     ↓                                 │
  [사람 승인] ── 필터 재조정 요청 시 ─────────┘
     │ 승인
     ↓
  [최종 저장: Spotify 플레이리스트 또는 JSON]
  ※ 전면 재시작 요청 시 역할 1로 복귀

- 공유 상태:
  class PlaylistState(TypedDict):
      context: str           # 역할 1이 쓰고 모두 읽음
      music_attrs: dict      # 역할 1 출력 → 역할 2 입력
      filters: dict          # 사용자 입력 → 역할 2 입력
      search_query: dict     # 역할 2 출력 → 검색·역할 3 입력
      candidate_songs: list  # Spotify 검색 결과 → 역할 3 입력
      playlist_draft: list   # 역할 3 출력 → 승인 게이트 입력
      approval_status: str   # 사람 입력: approved / revise_filter / restart
      revision_note: str     # 재조정 시 사람의 피드백 메모
  ※ 각 역할은 자신에게 필요한 필드만 읽는다

- 사람 승인 지점:
  - 위치: 역할 3 완료 직후, Spotify 저장 직전  
  - 사람이 선택하는 옵션: ① 승인(저장 진행) ② 필터 재조정(역할 2 재실행) 
    ③ 전면 재시작(역할 1 복귀)
  - 자동화하지 않는 이유: 취향은 정답이 없고 같은 조건이라도 그날 기분에 따라 
    결과가 달라지기 때문에 최종 판단은 반드시 사람이 해야 한다

- 멀티 에이전트가 더 나아질 것이라 보는 근거:
  - 상황 해석, 필터 병합, 순서 배치의 책임이 서로 다르고 각 단계의 입력과 출력이. 분리된다
  - 필터 재조정 루프에서 역할 1은 재실행하지 않아도 되므로 불필요한 LLM 호출이. 줄고 응답이 빠르다
  - 역할 3이 BPM 연속성·아티스트 다양성 검증을 전담하므로 역할 2는 검색. 파라미터 생성에만 집중할 
    수 있다

- 멀티 에이전트를 쓰면 안 되는 조건:
  - Spotify API 없이 LLM이 곡명을 직접 생성하는 경우 — 검색과 구성 역할 분리의. 의미가 없다
  - 상황이 단 1~2개로 고정되어 매핑 테이블이 필요 없는 경우 — 구조화된 단일. 프롬프트로 충분하다
  - 필터 없이 상황만으로 플레이리스트를 만드는 1회성 사용이라면 — 역할 2가 
    존재할 이유가 없다
  - 역할을 나눠도 역할 2와 역할 3의 출력이 거의 같아진다면 — 잘못 나눈 것이라는 신호다
