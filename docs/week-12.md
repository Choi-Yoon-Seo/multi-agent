# Week 12. 내 에이전트 1차 개선: 중간 정보와 판단 기준

> 11주차에 만든 `my_agent.py`를 개선한다.
일정공지 에이전트는 공통 예시일 뿐이다. 학생은 각자 정한 문제와 입력 자료에 맞게 서로 다른 에이전트를 구축한다.
이번 주의 핵심은 모든 에이전트에 공통으로 필요한 "입력 분석 -> 중간 정보 -> 판단/처리 -> 결과 저장" 흐름을 코드에 분명히 만드는 것이다.
> 

---

## 학습 목표

- 내 에이전트가 어떤 유형인지 설명한다.
- 입력 자료에서 다음 단계에 필요한 중간 정보를 만든다.
- 중간 정보를 리스트와 딕셔너리 형태로 정리한다.
- 내 에이전트의 판단 기준이나 처리 기준을 코드에 드러낸다.
- 결과를 주제에 맞는 Markdown 파일로 저장한다.
- LLM이나 외부 도구를 사용할 경우, 기본 실행을 해치지 않도록 fallback을 둔다.
- 코딩에이전트에게 작은 단위로 수정 요청하는 방법을 익힌다.

---

## 이번 주 핵심 원칙

1. 새 기능은 하나씩 추가한다.
2. 코드 전체를 다시 만들게 하지 않는다.
3. 11주차 구조를 유지한 채 입력 분석, 판단/처리, 저장 기능을 개선한다.
4. 모든 에이전트가 "표 추출 에이전트"일 필요는 없다.
5. 완벽한 자연어 처리를 목표로 하지 않는다.
6. LLM과 외부 도구는 보조 수단이다. 사용하더라도 실패 시 기본 실행은 가능해야 한다.
7. 실행 결과 파일을 남긴다.

## 12주차 공통 개선 흐름

에이전트 주제가 달라도 이번 주에 확인할 공통 흐름은 같다.

```
입력 자료
   ↓
중간 정보 만들기
   ↓
판단 기준 또는 처리 기준 적용
   ↓
결과 문서 생성
   ↓
실행 결과 확인
```

일정공지 예시는 이 흐름의 한 사례이다.

| 공통 단계 | 일정공지 예시 | 다른 에이전트에서 바꿀 것 |
| --- | --- | --- |
| 입력 자료 | 학과 공지문 | 글, 코드, 질문, 파일 목록, 데이터, 사용 조건 |
| 중간 정보 | 제목, 대상, 날짜, 준비물 | 내 에이전트가 판단에 사용할 핵심 정보 |
| 판단/처리 | 학년별 분류 | 점검, 평가, 생성, 검색, 변환, 실행 계획 |
| 결과 문서 | `output.md` 표 | 보고서, 안내문, 답변 로그, 초안, 처리 결과 |
| 검토 기준 | 날짜 누락, 대상 불명확 | 근거 부족, 조건 미반영, 실패 처리, 위험한 단정 |

| 출력 형식 | 적합한 경우 | 파일 예시 |

## 12.3 중간 정보 구조 정하기

중간 정보는 다음 단계가 읽을 수 있는 형태여야 한다. 가장 단순한 구조는 딕셔너리의 리스트이다.

```python
[
    {
        "item": "예시 항목",
        "category": "분류 또는 판단 기준",
        "detail": "핵심 정보",
        "evidence": "입력에서 확인한 근거",
        "note": "주의할 점",
    }
]
```

키 이름은 에이전트 유형에 맞게 바꾼다.

| 유형 | dict key 예시 |
| --- | --- |
| 정보 추출형 | `title`, `date`, `target`, `task`, `source` |
| 검토형 | `location`, `issue`, `severity`, `evidence`, `suggestion` |
| 생성형 | `topic`, `audience`, `constraints`, `outline`, `draft_point` |
| 질의응답형 | `question`, `intent`, `source`, `answer_status`, `answer` |
| 도구 실행형 | `item`, `action`, `status`, `result`, `error` |
| 의사결정 보조형 | `candidate`, `criterion`, `score`, `reason`, `risk` |

핵심은 함수 이름이 아니라 앞 단계의 결과가 다음 단계의 입력으로 이어지는 것이다.

## 12.4 코딩에이전트 요청 1: 입력 분석 함수 개선

11주차 코드에 `extract_facts` 함수가 있다면 그 함수를 개선해도 된다. 다른 이름을 쓰고 싶다면 역할이 드러나는 이름으로 바꾼다.

| 공통 역할 | 일정공지 예시 함수 | 다른 이름 예시 |
| --- | --- | --- |
| 입력 분석 | `extract_facts` | `analyze_input`, `parse_items`, `inspect_text` |
| 판단/처리 | `classify_items` | `review_items`, `score_items`, `plan_actions` |
| 출력 생성 | `write_output` | `write_report`, `write_answer`, `write_draft` |
| 저장 | `save_markdown_table` | `save_report`, `save_log`, `save_result` |

먼저 입력 분석 함수만 고친다.

```
my_agent.py를 읽어줘.
이번에는 입력 분석 함수만 개선할 거야.

내 에이전트 유형:
- 정보 추출형 / 검토형 / 생성형 / 질의응답형 / 도구 실행형 / 의사결정 보조형 / 기타

목표:
- 입력 자료에서 다음 단계에 필요한 중간 정보를 dict로 뽑는다.
- 결과는 dict의 list로 반환한다.
- dict의 key는 내 에이전트 유형에 맞게 4~6개 정도 사용한다.
- 이 단계에서는 외부 패키지를 쓰지 않는다.
- 판단/처리 함수와 출력 함수의 큰 구조는 아직 바꾸지 않는다.

먼저 수정 계획만 말해줘. 아직 파일을 수정하지 마.
```

계획을 확인한 뒤 실행한다.

```
좋아. 이제 입력 분석 함수만 수정해줘.
수정 후 main에서 중간 정보를 보기 좋게 출력해줘.
```

## 12.5 판단 또는 처리 기준 개선

두 번째 단계는 중간 정보를 보고 판단하거나 처리하는 함수이다. 정보 추출형에서는 분류가 자연스럽지만, 다른 유형에서는 점검, 평가, 생성 계획, 답변 가능 여부 판단, 실행 순서 결정이 될 수 있다.

| 유형 | 판단/처리 예시 |
| --- | --- |
| 정보 추출형 | 항목을 대상, 날짜, 주제, 중요도별로 묶는다 |
| 검토형 | 문제점을 심각도별로 나누고 근거가 있는지 확인한다 |
| 생성형 | 입력 조건에 맞게 목차나 문단 순서를 정한다 |
| 질의응답형 | 바로 답할 수 있는 질문과 추가 확인이 필요한 질문을 나눈다 |
| 도구 실행형 | 어떤 항목에 어떤 작업을 적용할지 정한다 |
| 의사결정 보조형 | 후보별 점수와 위험 요소를 비교한다 |

코딩에이전트 요청:

```
판단/처리 함수를 개선해줘.

규칙:
- 입력 분석 함수가 반환한 dict list를 입력으로 받는다.
- 내 에이전트 유형에 맞는 판단 또는 처리 결과를 반환한다.
- 복잡한 AI 판단은 하지 말고 문자열 포함 여부나 간단한 조건으로 처리한다.
- 판단 기준을 코드 안에서 읽기 쉬운 변수명이나 주석으로 드러내줘.
- 기존 출력 함수의 큰 구조는 아직 바꾸지 않는다.

먼저 수정 계획만 말해줘. 아직 파일을 수정하지 마.
```

## 12.7 LLM 보조 구조

LLM 보조 구조는 중급 구현이다. 사용하지 않아도 규칙 기반 기본 실행은 가능해야 한다. 사용하는 경우에도 LLM이 모든 책임을 가져가면 안 된다.

권장 구조:

```python
USE_LLM = False

def analyze_input(text):
    if USE_LLM:
        return analyze_with_groq(text)
    return analyze_with_rules(text)
```

코딩에이전트 요청:

```
입력 분석 함수에 LLM 보조 구조를 추가해줘.

요구사항:
- 기존 규칙 기반 분석은 analyze_with_rules 함수로 유지해줘.
- Groq API를 쓰는 analyze_with_groq 함수를 추가하고 구조를 나눠줘.
- USE_LLM = False가 기본값이어야 해.
- GROQ_API_KEY는 환경변수에서 읽고, 코드에 직접 쓰지 마.
- API 키가 없거나 호출에 실패하면 규칙 기반 분석으로 돌아가게 해줘.
- 아직 LangGraph는 사용하지 마.

먼저 수정 계획만 말해줘. 아직 파일을 수정하지 마.
```

기존 함수 이름이 `extract_facts`라면 `analyze_input` 대신 `extract_facts`에 같은 구조를 적용해도 된다.

## 12.8 보조 도구 또는 외부 자원 1개 연결하기

보조 도구는 에이전트의 판단이나 처리를 돕는 수단이다. 웹 API만 뜻하지 않는다. 로컬 파일, CLI 명령, MCP 도구, 검색 결과 파일, 규칙 검사 함수처럼 에이전트가 사용할 수 있는 보조 수단을 포함한다.

함수 자체는 외부 도구가 아니지만, 도구나 외부 자원을 에이전트 코드에 연결하는 가장 단순한 인터페이스이다. 따라서 API 호출, 파일 읽기, CLI 실행, 규칙 검사는 코드 곳곳에 흩어 쓰지 말고 작은 함수로 감싼다.

Skills는 조금 다르다. Skills는 보통 `my_agent.py`가 실행 중에 호출하는 도구라기보다, 코딩에이전트가 개발 작업을 할 때 따르는 절차와 규칙이다. 따라서 과제 코드의 도구로 쓰기보다는, 코딩에이전트에게 작업 방식을 알려 주는 개발 보조 자료로 이해한다.

| 도구/자원 종류 | 예시 | 코드에서 감싸는 함수 예시 |
|---|---|
| 웹 API | 날씨 API, 도서 검색 API, 환율 API | `fetch_weather()`, `search_books()` |
| 로컬 파일 | CSV, JSON, txt, 수업자료 폴더 | `load_reference_data()` |
| CLI 명령 | `git`, `python`, `pandoc`, `ffmpeg`, `playwright-cli` | `run_cli_tool()` |
| MCP 도구 | GitHub MCP, 파일 검색 MCP, 브라우저 MCP | `fetch_from_mcp()` |
| 검색 결과 파일 | 미리 저장한 검색 결과, RSS 결과 | `load_search_results()` |
| 규칙 검사 함수 | 금지어 검사, 점수 계산, 형식 검사 | `check_with_rules()` |

에이전트 유형별로는 다음처럼 고를 수 있다.

| 유형 | 보조 도구 또는 외부 자원 예시 |
| --- | --- |
| 정보 추출형 | 기관 공지 URL, 로컬 문서 파일, CSV 목록, 파일 검색 도구 |
| 검토형 | 규정 파일, 스타일 가이드, 테스트 결과 파일, 규칙 검사 함수 |
| 생성형 | 키워드 목록, 템플릿 파일, 예시 문서, 로컬 자료 폴더 |
| 질의응답형 | FAQ 파일, 수업자료 폴더, 규정 문서, 검색 결과 파일 |
| 도구 실행형 | 파일 시스템, 변환 대상 폴더, CLI 명령, MCP 도구 |
| 의사결정 보조형 | 가격표, 점수표, 환율/날씨/통계 API, 계산 함수 |

---

## 에이전트의 정의

**에이전트란, 입력을 분석하여 구조화된 중간 정보를 만들고, 명시적 기준에 따라 판단·처리한 뒤, 결과를 생성하는 프로그램이다.**

핵심: "LLM에 맡기는 것"이 아니라 "내 코드가 판단하는 것"이다.

---

## 필수 기준 4가지

이 4가지가 모두 코드에 존재해야 에이전트라 볼 수 있다.

| # | 기준 | 확인 방법 | 없으면? |
| --- | --- | --- | --- |
| 1 | **구조화된 중간 표현** | 입력을 dict, list 등 코드가 다룰 수 있는 구조로 변환하는 함수가 있다 | 입력을 그대로 LLM에 전달하는 것은 에이전트가 아님 |
| 2 | **명시적 판단 기준** | 조건문, 규칙 변수, 점수 체계가 코드에 드러난다 | "알아서 판단해줘"를 LLM에 위임하면 에이전트가 아님 |
| 3 | **단계 분리** | 입력 분석 / 판단·처리 / 출력 생성이 별도 함수로 존재한다 | 하나의 프롬프트로 처음부터 끝까지 처리하면 에이전트가 아님 |
| 4 | **자율적 결정** | 중간 정보에 따라 에이전트가 다른 행동을 선택한다 (조건 분기) | 입력이 바뀌어도 항상 같은 경로로 실행되면 에이전트가 아님 |

---

## 강화 기준 5가지

있으면 더 높은 수준의 에이전트이다. 12주차에서는 5~6번을 목표로 한다.

| # | 기준 | 설명 |
| --- | --- | --- |
| 5 | **도구 사용 (Tool Use)** | 외부 API, 파일, 검색 등을 호출해 정보를 보강한다 |
| 6 | **Fallback / 견고성** | 도구나 LLM이 실패해도 규칙 기반으로 기본 동작을 유지한다 |
| 7 | **반복·피드백 루프** | 결과를 평가하고, 불충분하면 재시도하거나 수정한다 |
| 8 | **메모리 / 상태 유지** | 이전 실행 결과나 대화를 기억해 다음 판단에 반영한다 |
| 9 | **멀티 스텝 계획** | 목표를 하위 작업으로 분해하고 실행 순서를 정한다 |

## "에이전트 아닌 것" 체크리스트

다음 중 하나라도 해당하면 에이전트라 보기 어렵다.

- [ ]  LLM 단일 호출로 끝남 (프롬프트 하나 → 결과 하나)
- [ ]  중간 정보가 없음 (입력이 가공 없이 그대로 출력 단계로 감)
- [ ]  판단 주체가 전적으로 LLM (코드에 판단 로직이 없음)
- [ ]  조건 분기 없음 (입력이 바뀌어도 항상 같은 경로)
- [ ]  결과의 근거를 코드에서 추적할 수 없음

### X — 에이전트가 아닌 설계

| 설계 | 이유 |
| --- | --- |

| 공지문 → "요약해줘" 프롬프트 → 출력 | 프롬프트 래퍼일 뿐 |
| 코드 → "리뷰해줘" 프롬프트 → 피드백 텍스트 | LLM이 모든 판단 수행 |
| 질문 → ChatGPT API 호출 → 답변 저장 | 챗봇 래퍼이지 에이전트 아님 |

### O — 에이전트인 설계

| 설계 | 이유 |
| --- | --- |
| 방 사진 → 가구 목록 추출 → 크기·위치 구조화 → 동선·채광 규칙 적용 → 배치안 생성 | 중간 정보 있음, 규칙 기반 판단, 단계 분리 |
| 공지문 → 항목 추출(날짜, 대상, 내용) → 학년별 분류 → 표 생성 | dict list 구조, 조건 분기, 결과 추적 가능 |
| 코드 → 함수별 분리 → 복잡도·네이밍 규칙 점검 → 심각도 분류 → 리뷰 리포트 | 명시적 기준, 단계별 처리 |
| 이력서 → 항목 추출 → 직무 요구사항과 대조 → 점수·피드백 생성 | 구조화, 대조 규칙, 조건별 출력 |

## "대신 일해주면 에이전트 아닌가?" - 자주 나오는 질문

### 질문

> "어쨌든 LLM이 내 대신 일을 수행하니까, 그것도 에이전트 아닌가?"
> 

사전적 의미로는 맞다. Agent는 라틴어 *agere*(행동하다)에서 왔고, "대리인"이라는 뜻이다. 이 정의대로면 계산기도, 세탁기도, ChatGPT도 에이전트이다.

### 왜 구분하는가

구분하는 이유는 **통제 가능성**과 **학습 가치** 때문이다.

### 1. 통제 가능성의 차이

|  | LLM에 통째로 맡김 | 에이전트 구조 |
| --- | --- | --- |
| 결과가 틀릴 때 | 프롬프트를 바꿔보는 수밖에 없음 | 어느 단계에서 틀렸는지 추적 가능 |
| 재현성 | 같은 입력도 매번 다른 결과 | 규칙 기반 부분은 동일 결과 보장 |
| 수정 범위 | "프롬프트 엔지니어링" | 특정 함수·조건만 수정 |

"왜 이 결과가 나왔는가"를 설명할 수 없으면, 그건 **도구를 사용한 것**이지 **설계한 것**이 아니다.

### 2. 비유

| 비유 | 에이전트 설계인가? |
| --- | --- |
| 변호사에게 "알아서 해주세요" 위임 | X — 위임이지 내가 설계한 시스템이 아님 |
| 업무 매뉴얼 + 판단 기준 + 보고 양식을 만들어서 직원에게 줌 | O — 시스템 설계 = 에이전트 구축 |

LLM에 "알아서 해줘"라고 하는 것은 전자이다.  **판단 구조를 설계**하는 것이 목적이다.

### 3. 학습 가치

"대신 일해주는 것"만으로 에이전트를 인정하면:

```python
response = openai.chat.completions.create(
    messages=[{"role": "user", "content": input_text}]
)
print(response)
```

이 코드에서 배우는 것은 **API 호출법**뿐이다. 에이전트에서 배워야 할 것 — 문제 분해, 중간 표현 설계, 판단 기준 명시, 실패 처리 — 은 전혀 없다.

### 결론

| 관점 | 에이전트인가? |
| --- | --- |

| AI 설계 관점 ("자율 시스템") | 내 코드에 판단 구조가 있어야 O |
| 이 수업의 학습 목표 | 학생이 설계한 로직이 보여야 O |

> "맞다, 넓은 의미로는 에이전트다. 하지만 **판단 구조를 가진** 에이전트가 올바른 에이전트 정의이다. 결과가 틀렸을 때 '프롬프트를 바꿔보자'가 아니라 '3번째 조건문을 고치자'라고 말할 수 있어야 한다."
