# Week 11. 바이브코딩으로 내 에이전트 v0 만들기

챕터별로 새로운 세션으로 시작하며 아래의 프롬프트로 시작한다.

```
이 프로젝트를 검토 이해한다. 오늘은 week-11.md 작업을 수행한다.
응답은 한국어로 하고 설명은 이해하기 쉽게 한다.
```

> 강의자료에서는 일정공지 에이전트를 공통 예시로 사용한다.
> 

> 학생은 같은 구조를 참고하여 본인이 선택한 주제의 에이전트 v0를 만든다.
> 

> 이번 주에는 LangGraph Supervisor, Groq API, 외부 API를 붙이지 않는다. Python 함수 3개를 에이전트 역할로 나누어 가장 쉬운 멀티에이전트 뼈대를 만든다.
> 

## 학습 목표

- 일정공지 에이전트 예시를 자기 주제의 에이전트 구조로 바꾼다.
- `context.md`와 `todo.md`를 코딩에이전트에게 맥락으로 제공한다.
- 바이브코딩 프롬프트를 사용해 `my_agent.py`의 첫 버전을 만든다.
- 코딩에이전트에게 “계획 먼저, 코드 나중” 방식으로 작업을 맡긴다.
- 규칙 기반 v0와 이후 LLM·외부 도구 확장의 차이를 구분한다.
- 실행 결과를 보고 다음 주 개선 항목을 `todo.md`에 남긴다.

---

## 이번 주 핵심 원칙

1. 일정공지 에이전트는 예시이고, 제출물은 본인의 에이전트이다.
2. 직접 처음부터 코드를 짜기보다, 코딩에이전트에게 명확한 요청을 준다.
3. 한 번에 완성된 프로그램을 요구하지 않고, 계획과 실행을 나누어 맡긴다.
4. LangGraph, RAG, 데이터베이스는 쓰지 않는다.
5. 외부 패키지 없이 Python 기본 코드로 시작한다.

## 이번 주 에이전트의 의미

이번 주에 만드는 에이전트는 LLM API를 호출하는 지능형 에이전트가 아니다. Python 함수 여러 개가 각각 역할을 맡는 **규칙 기반 함수형 에이전트**이다.

즉, 딥러닝 모델이 판단하는 구조가 아니라 다음 요소로 동작한다.

- 문자열 포함 여부
- `if` 조건문
- 리스트와 딕셔너리
- 정해진 출력 템플릿
- 사람이 정한 규칙

이 방식은 단순하지만 에이전트 구조를 이해하기에 좋다. 나중에 Groq API, 외부 도구, LangGraph를 붙이더라도 역할 분리 구조는 그대로 이어진다.

11~14주차 프로젝트는 다음 단계로 확장한다.

| 주차 | 기본 목표 | 선택 확장 |
| --- | --- | --- |
| 11주차 | 규칙 기반 함수 에이전트 v0 | 없음 |
| 12주차 | 추출과 분류 개선, `output.md` 저장 | Groq API 또는 외부 도구 1개 사용 |
| 13주차 | 최종 출력과 검토자 추가 | LLM 검토자 또는 도구 결과 반영 |
| 14주차 | 최종 정리와 발표 준비 | 기본형·중급형·확장형·심화형 중 구현 수준 명시 |

## 11.2 일정공지 예시를 내 에이전트로 바꾸기

강의자료의 공통 예시는 일정공지 에이전트이다.

```
[공지 입력]
      ↓
extract_facts()
      ↓
classify_schedule()
      ↓
write_guide()
      ↓
print result
```

학생은 이 구조를 자기 주제로 바꾼다.

```
[내 입력 자료]
      ↓
extract_facts()
      ↓
classify_items()
      ↓
write_output()
      ↓
print result
```

아래 표를 채우면 일정공지 예시를 내 에이전트로 바꾸기 쉽다.

| 일정공지 에이전트 예시 | 내 에이전트에서 바꿀 것 |
| --- | --- |
| 공지문 | 내가 처리할 입력 자료 |
| 제목, 대상, 날짜, 준비물 | 내 주제에서 뽑을 핵심 정보 |
| 학년 또는 사용자 유형별 분류 | 내 주제의 분류 기준 |
| 사용자별 안내문 | 내 에이전트의 최종 출력 |
| `schedule_agent.py` | `my_agent.py` |

예시는 다음과 같다.

| 주제 | 입력 | 추출 | 분류 | 출력 |
| --- | --- | --- | --- | --- |
| 일정공지 | 학과 공지 | 날짜, 대상, 준비물 | 학년별 | 해야 할 일 안내 |
| 독서 추천 | 책 목록 | 장르, 난이도, 관심사 | 독자 유형별 | 추천 목록 |
| 운동 루틴 | 운동 목표 | 목표, 시간, 장비 | 초급/중급/고급 | 루틴 안내 |
| 여행 계획 | 여행 조건 | 날짜, 예산, 장소 | 일정별 | 여행 일정표 |

## 11.3 내 에이전트 계획 요청 프롬프트

먼저 코딩에이전트에게 내 주제를 설명하고 계획만 받는다.

```
context.md와 todo.md를 읽어줘.
docs/week-11.md의 일정공지 에이전트 예시를 참고해서
내가 선택한 주제의 에이전트 v0를 만들 거야.

내 에이전트 정보는 다음과 같아.
- 에이전트 이름:
- 해결하려는 문제:
- 입력 자료:
- 추출할 핵심 정보:
- 분류 기준:
- 최종 출력:
- 만들 파일: my_agent.py

아직 코드를 작성하지 말고,
이 에이전트를 함수 3개 구조로 만들 계획만 제시해줘.

조건:
- 외부 패키지는 사용하지 말 것
- LangGraph, RAG, 데이터베이스는 사용하지 말 것
- Groq API나 외부 API는 아직 붙이지 말 것
- Python 기본 코드로 만들 것
- 앞 단계 출력이 다음 단계 입력으로 들어가게 할 것
```

계획을 볼 때는 함수 이름보다 역할 분리가 보이는지 확인한다.

| 확인할 점 | 통과 기준 |
| --- | --- |
| 파일 범위 | `my_agent.py` 하나만 만든다 |
| 기술 범위 | 외부 패키지를 쓰지 않는다 |
| 역할 분리 | 추출, 분류, 출력 작성 역할이 나뉜다 |
| 실행 흐름 | 앞 단계 출력이 다음 단계 입력으로 들어간다 |

## 11.6 코딩에이전트에게 코드 확인시키기

코딩에이전트가 만든 코드 전체를 사람이 먼저 완벽히 이해하려고 하지 않는다. 오늘은 코딩에이전트에게 세 부분을 점검하게 하고, 사람은 보고 결과가 맞는지 확인한다.

```
my_agent.py를 읽고 아래 세 가지를 점검해줘.

아직 코드를 수정하지 말고, 점검 결과만 표로 알려줘.

1. 함수가 최소 3개로 나뉘어 있는가?
   - 핵심 정보 추출 역할
   - 분류 역할
   - 최종 출력 작성 역할

2. 앞 단계 출력이 다음 단계 입력으로 들어가는가?
   - 예: facts = extract_facts(SAMPLE_INPUT)
   - 예: grouped = classify_items(facts)
   - 예: result = write_output(grouped)

3. 중간 결과가 출력되는가?
   - 예: print("=== facts ===")
   - 예: print(facts)

표에는 다음 열을 넣어줘.
- 확인 항목
- 통과 여부
- 코드에서 찾은 근거
- 고쳐야 할 점
```

점검 결과에서 실패한 항목이 있으면, 그 항목만 고치게 한다.

```
점검 결과에서 실패한 항목만 고쳐줘.

조건:
- my_agent.py의 전체 구조는 유지할 것
- 새 기능을 추가하지 말 것
- 함수 역할 분리, 단계별 입력/출력 연결, 중간 출력 중 부족한 부분만 수정할 것
- 수정 후 어떤 항목이 통과하게 되었는지 짧게 설명할 것
```

## 11.7 실행과 오류 수정 프롬프트

터미널에서 실행한다.

```bash
python my_agent.py
```

실행이 안 되면 오류 메시지를 그대로 복사해서 준다.

```
다음 오류가 나왔어.

(여기에 오류 메시지 붙여넣기)

my_agent.py의 전체 구조는 유지해줘.
새 기능을 추가하지 말고, 오류만 고쳐줘.
수정한 이유를 초보자가 이해할 수 있게 짧게 설명해줘.
```

실행은 되지만 결과가 이상하면 출력 예시와 기대를 함께 준다.

```
실행은 되지만 출력이 이상해.

현재 출력:
(여기에 현재 출력 붙여넣기)

내가 기대한 출력:
- 입력 자료별 핵심 정보가 보여야 함
- 내가 정한 분류 기준에 따라 나뉘어야 함
- 마지막에는 최종 출력이 Markdown처럼 보여야 함

코드 구조는 유지하고, 출력 형식만 알아보기 쉽게 고쳐줘.
```
