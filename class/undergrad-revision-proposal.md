# 학부생용 축약 운영안 제안

## 목적

이 문서는 현재 `class` 폴더의 주차별 강의안을 기준으로, 학부 정규 15주 과목에서 현실적으로 운영 가능한 축약안을 제안한다.

핵심 원칙은 다음과 같다.

- 한 학기 안에 모든 최신 에이전트 개념을 다 가르치려 하지 않는다
- 학생이 실제로 구현하고 제출할 수 있는 범위만 필수로 둔다
- 운영·배포·평가 프레임워크는 선택 심화 또는 읽기 자료로 내린다
- 과목 목표를 "작은 자동화 에이전트를 설계하고 구현할 수 있다" 수준으로 맞춘다

---

## 권장 과목 수준

- 권장 대상: Python 기초, Git/GitHub 기초, HTTP/API 호출 경험이 있는 3-4학년 학부생
- 비권장 대상: 프로그래밍 입문 직후 수강생, Git과 가상환경을 거의 써보지 않은 학생

현재 `class` 자료는 `docs`보다 수업 친화적으로 재작성되어 있으나, 후반부 난도는 여전히 높다. 따라서 다음과 같이 `필수 트랙`과 `심화 트랙`을 분리하는 것이 적절하다.

---

## 학습 목표 재정의

기존 목표를 그대로 유지하면 범위가 과도해진다. 학부 과목에서는 아래 수준으로 조정하는 것을 권장한다.

1. AI 코딩 도구를 사용할 때 사람의 역할과 검증 책임을 설명할 수 있다
2. 재현 가능한 개발 환경을 구성하고 작은 실습 프로젝트를 관리할 수 있다
3. 기존 MCP 서버를 연결하여 간단한 자동화 작업을 수행할 수 있다
4. 간단한 외부 API를 래핑한 최소 MCP 서버를 구현할 수 있다
5. 도구 호출 에이전트와 기본 워크플로우를 구현할 수 있다
6. RAG 또는 승인 워크플로우 중 한 가지 심화 주제를 선택해 소규모 프로젝트로 확장할 수 있다

## 최종 프로젝트 범위 제안

학부 과목의 최종 프로젝트는 아래 중 하나 수준이면 충분하다.

- 작은 외부 API를 MCP 서버로 래핑하고 간단한 에이전트와 연결
- 문서 5~10개 규모의 소형 RAG 질의응답 시스템 구현
- 승인 단계가 포함된 간단한 워크플로우 구현

다음은 권장하지 않는다.

- 멀티에이전트 필수화
- GraphRAG 필수화
- 실제 운영 배포와 자동 롤백 구현
- 복잡한 감사 로그 무결성 구현

## 결론

현재 `class` 자료는 `docs`보다 훨씬 수업 친화적이다. 다만 후반부의 심화 내용을 모두 필수로 유지하면 학부생 다수가 개념 이해는 하되 구현 경험은 얕게 가져갈 가능성이 높다.

따라서 학부 정규과목에서는 다음 원칙을 권장한다.

- 앞부분은 실습 중심으로 충분히 깊게 간다
- 후반부는 핵심 개념만 남기고 운영론은 과감히 선택화한다
- 프로젝트는 "작지만 끝까지 완성"하는 경험에 집중한다

이 기준으로 조정하면 `class` 폴더는 학부 15주 과목의 실질적인 기반 자료로 사용할 수 있다.
