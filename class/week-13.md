# Week 13. 최종 프로젝트 설계

## 학습 목표

- 최종 프로젝트의 문제를 명확히 정의한다
- 입력, 출력, 처리 흐름을 문서로 정리한다
- 구현 범위를 현실적으로 제한한다
- 검증 방식과 발표 계획을 함께 설계한다

---

## 이번 주 운영 원칙

- 큰 아이디어보다 완성 가능한 범위를 잡는 것이 중요하다
- 운영 배포보다 문제 정의, 흐름 설계, 위험 요소 정리가 우선이다
- 최종 프로젝트는 "작지만 끝까지 동작하는 것"을 목표로 한다

---

## 프로젝트 권장 범위

다음 중 하나 수준을 권장한다.

1. 작은 외부 API를 래핑한 MCP 서버
2. 도구 호출 에이전트
3. LangGraph 기반 생성-검증-수정 워크플로우
4. 문서 5~10개 규모의 소형 RAG 시스템
5. 승인 단계가 포함된 간단한 자동화 도구

권장하지 않는 범위:

- 멀티에이전트 필수화
- GraphRAG 구현
- 실제 운영 배포
- 복잡한 비용 최적화와 관측 시스템

---

## 13.1 문제 정의

다음 질문에 답할 수 있어야 한다.

- 무엇을 자동화하려는가
- 누가 사용하는가
- 현재 불편은 무엇인가
- 자동화 결과가 어떻게 보이면 성공인가

---

## 13.2 입력과 출력 정의

프로젝트 설계에서 가장 먼저 고정해야 할 것은 입력과 출력이다.

예시:

- 입력: 도시 이름
- 출력: 날씨 요약 텍스트 파일

또는

- 입력: 질문 + 문서 집합
- 출력: 출처가 포함된 답변

---

## 13.3 처리 흐름 설계

아래 정도만 정리해도 충분하다.

1. 입력을 받는다
2. 필요한 도구 또는 문서를 선택한다
3. 처리한다
4. 검증한다
5. 결과를 저장하거나 출력한다

복잡한 도식보다 한 페이지 안에 설명 가능한 흐름이 좋다.

---

## 13.4 위험 요소와 검증

학생 프로젝트라도 위험 요소를 미리 적어 보는 습관이 중요하다.

예시:

- 입력 오류
- 외부 API 실패
- 잘못된 문서 검색
- 검증 없는 출력

각 위험에 대해 최소 대응 하나를 적는다.

- 입력 검증
- 타임아웃
- 출처 표시
- 승인 단계

---

## 13.5 발표 계획

최종 발표는 아래 순서로 준비하도록 한다.

1. 문제 정의
2. 구조 설명
3. 실행 시연
4. 한계와 개선점

발표는 구현보다 설명이 약해지는 경우가 많으므로, 이번 주부터 시연 흐름을 미리 정리한다.

---

## 제출물

- 프로젝트 제안서 1부
- 입력/출력 정의
- 흐름도 또는 표
- 구현 계획
- 위험 요소와 대응 방법

---

## 체크리스트

- 주제가 한 학기 안에 끝날 크기인가
- 입력과 출력이 명확한가
- 도구 또는 데이터가 실제로 준비 가능한가
- 실패 시 어떻게 할지 적었는가
- 발표 시연이 가능한가
