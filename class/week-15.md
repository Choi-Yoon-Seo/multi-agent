# Week 15. 최종 프로젝트 발표 (기말고사 대체)

> 학부 운영안 — 발표 + 구두 평가
> 기말고사를 폐지하고, 발표·질의응답으로 통합 평가한다

---

## 학습 목표

- 자신이 만든 멀티에이전트 시스템을 *문제 → 아키텍처 → 데모 → 한계* 순서로 설명한다
- 5분 질의응답에서 자기 시스템의 한계를 정확히 답한다
- 다른 학생의 사례를 보고 자기 설계 선택을 돌아본다

---

## 운영 방식

### 발표 형식

- **1인당 10분 발표 + 5분 질의응답**
- 발표 자료: 슬라이드 + 데모 (라이브 또는 영상)
- 질의응답: 교수 1~2개 + 동료 1개 이상 (자유)

### 시간 배분 (1회차·2회차)

| 시점 | 내용 |
|---|---|
| 1회차 (90분) | 발표 1부 — 6명 내외 (발표 60분 + 질의응답 30분) |
| 2회차 (90분) | 발표 2부 — 잔여 + 종합 강평 + 우수 사례 공유 |

---

## 발표 권장 구성 (10분)

| 시간 | 내용 |
|---|---|
| 0~2분 | 문제 정의와 동기 — 왜 단일 에이전트로 안 되는가 |
| 2~5분 | 시스템 아키텍처 — Supervisor·역할 분담·통합 요소 |
| 5~8분 | 데모 시연 — 정상 1, 실패/경계 1 |
| 8~10분 | 한계, 알려진 실패, 다음 단계 |

**라이브 시연이 막히면 백업 영상으로 즉시 전환한다** (시간 깎이지 않음).

---

## 평가 기준 (40% 비중)

| 항목 | 비중 | 무엇을 보는가 |
|---|---|---|
| 설계서 (13주차 제출분) | 5% | 문제·아키텍처·일정·위험 정리 |
| MCP 서버 구현 | 20% × 0.5 = 10% | 도구·리소스, 에러 처리 |
| 멀티에이전트 워크플로우 | 30% × 0.5 = 15% | Supervisor·핸드오프·공유 메모리 |
| 통합 요소 (RAG·HITL·배포 중 2개+) | 20% × 0.5 = 10% | 실제 데모에서 작동 |
| 발표 + 구두 평가 | 20% × 0.5 = 10% | 논리 구성, 시연 완성도, 질의응답 |
| 코드 품질 | 10% × 0.5 = 5% | 구조, 문서, Git 이력 |

> 환산표: 총 40점 만점. 위 비율은 강의계획서 “최종 프로젝트 40%” 안에서의 세부 가중.

---

## 질의응답에서 자주 나오는 질문 (대비용)

학생이 미리 답을 준비해 두면 좋다.

1. 단일 에이전트로 만들지 *않은* 이유는 무엇인가
2. 가장 자주 라우팅이 잘못된 사례는 무엇인가
3. 환각이 한 번이라도 답변에 들어간 적이 있는가, 어떻게 잡는가
4. HITL 승인을 *생략하면* 어떤 일이 벌어지는가
5. 비용이 가장 큰 노드는 어디고, 어떻게 줄이겠는가
6. 같은 시스템을 동료가 자기 노트북에서 돌릴 수 있는가
7. 이 시스템을 한 학기 더 발전시킨다면 무엇을 추가하겠는가

답이 “모르겠다”라면 *그렇다고 답한다*. 모르는 것을 안다고 답하면 감점이 더 크다.

## 제출물 (발표 전까지)

- 최종 코드 (GitHub 링크)
- 발표 슬라이드 (PDF 또는 키노트)
- 데모 영상 (라이브 시연이 잘 돼도 *백업으로 반드시 제출*)
- 1쪽 회고 — 가장 잘 된 설계 / 가장 어려웠던 부분 / 다시 한다면 바꿀 것

---

## 한 학기 핵심 정리

이 과목이 끝난 시점에 학생이 가져가야 할 것은 다음 다섯 가지다.

1. **AI가 코드를 써도 사람의 검증 책임은 사라지지 않는다.** 11주차 GitHub Copilot 실습이 이 원칙의 가장 분명한 예였다.
2. **재현 가능한 실행 환경이 먼저다.** 2주차에 만든 가상환경과 13주차의 Docker가 여기에 해당한다.
3. **멀티에이전트는 “여러 모델을 부르는 일”이 아니라 “책임을 분리하고 검증하는 일”이다.** 9~10주차에서 단일 vs 멀티 비교로 본 그대로다.
4. **워크플로우는 명시적으로 설계할수록 디버깅이 쉽다.** LangGraph의 Supervisor 패턴과 LangSmith 트레이스가 이를 가능하게 했다.
5. **신뢰성은 “좋은 답을 많이 만드는 일”이 아니라 “틀릴 수 있음을 다루는 일”이다.** 12주차의 출처 검증 게이트와 13주차의 HITL 승인이 그 핵심 도구였다.

---

## 마무리

이 과목의 목표는 최신 유행 개념을 많이 훑는 것이 아니라, **작은 멀티에이전트 시스템을 끝까지 설계하고 구현하고 설명하는 경험**을 만드는 것이었다.

오늘 발표하는 시스템은 학기 시작 시점의 “MCP 서버 add 도구”에서 출발해 *Supervisor + RAG + HITL + Docker*까지 도달한 결과다. 다음 학기, 다음 직장, 다음 연구실에서 멀티에이전트 시스템을 만들 일이 생겼을 때, 오늘 발표한 자기 시스템이 출발점이 된다.
