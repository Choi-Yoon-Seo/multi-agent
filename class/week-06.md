## 6.1 왜 LangGraph가 필요한가

- ReAct 방식은 간단한 도구 호출에는 적합하다
- 하지만 다음 상황에서는 흐름 제어가 필요하다
  - 검증 결과에 따라 다른 경로로 가야 할 때
  - 실패 시 다시 시도해야 할 때
  - 여러 단계를 명시적으로 관리해야 할 때

LangGraph의 핵심 장점:

- 상태를 코드로 정의할 수 있음
- 분기 조건을 코드로 통제할 수 있음
- 반복 종료 조건을 명확히 넣을 수 있음

---

## 6.2 상태(State) 모델

- 워크플로우 전체에서 공유하는 데이터를 상태로 둔다
- 각 노드는 상태를 읽고 일부를 갱신한다

예시 상태:

```python
class WorkflowState(TypedDict):
    draft: str
    feedback: list[str]
    revision_count: int
```

이번 주에는 복잡한 메모리 체계보다 "현재 작업이 어디까지 왔는지"를 저장하는 용도로 이해하면 충분하다.

---

## 6.3 조건 분기

- 검증 결과가 좋으면 종료
- 부족하면 수정 단계로 이동

예시:

```python
def should_continue(state: WorkflowState) -> str:
    if state["revision_count"] >= 3:
        return "end"
    return "revise" if state["feedback"] else "end"
```

핵심은 두 가지다.

- 분기 조건을 사람이 읽을 수 있게 작성할 것
- 무한 루프가 생기지 않도록 종료 조건을 넣을 것

---

## 6.4 반복 루프

이번 주 대표 패턴:

1. 초안 생성
2. 검증
3. 부족하면 수정
4. 다시 검증
5. 최대 3회 후 종료

이 패턴을 구현하면 학생들은 "에이전트 워크플로우를 제어한다"는 감각을 얻을 수 있다.

---

## 6.5 실패 복구의 기본

실패 복구는 운영 수준의 복잡한 전략보다 아래 수준이면 충분하다.

- 모델 호출 실패 시 오류 메시지 저장
- 재시도 횟수 제한
- 실패가 반복되면 종료 또는 사람 확인으로 전환

이번 주에는 "완벽한 복구"보다 "실패를 숨기지 않는 구조"가 더 중요하다.

---

## 6.6 실습: 초안 생성 → 검증 → 수정

### 실습 목표

- LangGraph로 최소 순환 워크플로우를 구현한다

### 권장 노드

- `generate_draft`
- `validate`
- `revise`

### 권장 상태 필드

- `draft`
- `feedback`
- `revision_count`

### 권장 종료 규칙

- 검증 통과 시 종료
- 수정 횟수 3회 초과 시 종료

### 학생이 직접 확인할 것

- 분기가 원하는 대로 동작하는가
- 수정 횟수가 누적되는가
- 종료 조건이 없으면 어떤 문제가 생기는가
